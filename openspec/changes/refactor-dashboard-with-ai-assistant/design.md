## Context

当前首页（Home.vue/Dashboard）展示的统计数据是静态的，与实际模块数据不同步。用户希望首页能成为真正的"工作台"，展示今日重要信息并提供快速操作入口。

同时，用户希望通过AI自然语言接口快速完成跨模块操作，避免频繁的界面导航和表单填写。这需要一个统一的AI助手系统，能够理解用户意图并调用对应模块的功能。

### 约束
- 必须复用现有模块的数据库表和API
- AI功能依赖用户在设置中配置的API
- 需要保持向后兼容，不影响现有功能
- 响应时间要求：AI处理 < 5秒，数据加载 < 1秒

### 相关模块
- Dashboard (首页)
- AI Assistant (新增)
- Notes, Calendar, Passwords, Bookmarks, Todos (数据提供方)
- Settings (AI API配置)

## Goals / Non-Goals

### Goals
- 首页展示实时、有用的信息
- 提供AI自然语言交互界面
- 支持通过AI快速创建各模块数据
- 提升用户操作效率90%
- 保持系统一致性和可维护性

### Non-Goals
- 不重新设计各模块的UI
- 不实现AI自动化流程（如自动生成日报）
- 不支持语音输入（可作为未来扩展）
- 不实现复杂的多步骤工作流
- 不修改现有数据库schema

## Decisions

### 决策1: 首页数据源 - 直接读取各模块数据库
**选择**: 直接从各模块的数据库表读取数据
**理由**: 
- 保证数据一致性
- 无需维护额外的聚合表
- 实时性最好

**备选方案**:
- ❌ 创建聚合视图 - 增加复杂度，数据同步问题
- ❌ 使用各模块的API - 跨组件通信复杂

### 决策2: AI意图识别 - 基于Prompt工程
**选择**: 使用结构化Prompt让AI返回JSON格式的意图和参数
**理由**:
- 无需训练模型
- 灵活可调整
- GPT-4等大模型有足够的理解能力

**Prompt示例**:
```
你是一个助手，需要解析用户指令并返回JSON格式。
支持的操作：create_note, create_event, create_password, create_bookmark, create_todo

用户: 帮我写一个滕王阁序的md文档
返回: {
  "intent": "create_note",
  "params": {
    "title": "滕王阁序",
    "type": "md",
    "content_request": "生成滕王阁序全文"
  }
}
```

**备选方案**:
- ❌ 正则匹配 - 覆盖不全，易出错
- ❌ 训练分类模型 - 成本高，维护难

### 决策3: 操作确认机制 - 先AI生成，用户确认后执行
**选择**: AI生成内容后展示给用户，确认后才写入数据库
**理由**:
- 避免AI错误导致脏数据
- 用户有最终控制权
- 符合用户心智模型

**流程**:
1. 用户输入指令
2. AI解析意图和参数
3. AI生成内容（如笔记正文）
4. 展示预览和确认按钮
5. 用户确认后调用模块API创建

### 决策4: 多轮对话 - 使用会话历史
**选择**: 在本地维护对话历史（最近10条），发送给AI保持上下文
**理由**:
- 支持追问和补充信息
- 提升用户体验
- 实现简单

**备选方案**:
- ❌ 无状态单次请求 - 用户体验差
- ❌ 后端会话管理 - 增加复杂度

### 决策5: 时间解析 - 使用JS库 + AI辅助
**选择**: 使用 `chrono-node` 或类似库解析时间，AI提供自然语言描述
**理由**:
- 准确性高
- 覆盖中文时间表达
- 降低AI幻觉风险

**示例**:
- "明天上午10点" -> Date对象
- "下周三下午2点" -> Date对象

## Architecture

### 组件架构
```
Home.vue (仪表板)
├── TodayOverview.vue (今日概览)
├── TodayTodos.vue (今日待办)
├── TodayEvents.vue (今日日程)
├── RecentNotes.vue (最近笔记)
├── QuickBookmarks.vue (常用网站)
└── PasswordSummary.vue (密码统计)

AIAssistant.vue (AI助手)
├── ChatDialog.vue (对话框)
├── MessageList.vue (消息列表)
├── InputBox.vue (输入框)
└── ConfirmAction.vue (操作确认)
```

### 数据流
```
用户输入 → AI API → 意图识别 → 参数提取 → 内容生成 → 用户确认 → 模块API → 数据库
```

### AI服务层
```javascript
// src/utils/ai/
intent.js      // 意图识别
parser.js      // 参数解析
api.js         // AI API封装
prompt.js      // Prompt模板
operations.js  // 操作调度
```

## Risks / Trade-offs

### 风险1: AI API成本
- **风险**: 频繁调用GPT-4费用较高
- **缓解**: 
  - 支持使用更便宜的模型（GPT-3.5）
  - 本地缓存常见意图
  - 限制每日调用次数

### 风险2: 意图识别错误
- **风险**: AI误解用户意图
- **缓解**:
  - 操作前展示预览
  - 支持一键撤销
  - 记录操作日志

### 风险3: 时间解析歧义
- **风险**: "下周五"等表达有歧义
- **缓解**:
  - 显示解析后的具体时间
  - 允许用户修正

### 风险4: 密码安全
- **风险**: AI可能记录用户密码
- **缓解**:
  - 使用自定义API避免OpenAI记录
  - 在Prompt中强调不记录敏感信息
  - 密码字段特殊处理

## Open Questions

1. **Q**: 是否需要支持批量操作（如"创建10个待办"）？
   **A**: 第一版支持待办的批量创建，其他模块暂不支持

2. **Q**: AI生成的内容质量如何保证？
   **A**: 通过精心设计的Prompt和用户确认机制

3. **Q**: 是否需要训练数据集？
   **A**: 不需要，使用Few-shot learning在Prompt中提供示例

4. **Q**: 如何处理API配额耗尽？
   **A**: 显示友好错误提示，引导用户检查配置或更换API

5. **Q**: 是否需要支持插件扩展？
   **A**: 第一版不支持，预留接口便于未来扩展
