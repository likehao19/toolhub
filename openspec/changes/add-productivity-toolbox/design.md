# 设计：一体化生产力工具箱

## Context
基于现有的 Vue3 + Vite + Tauri 模板，扩展为完整的本地优先生产力工具箱。需要整合多个功能模块，支持插件化扩展，并保持代码结构清晰、可维护。

## 目标 / 非目标

### 目标
- 本地优先：所有数据存储在本地，用户完全控制
- 插件化架构：支持功能模块扩展
- 统一体验：一致的 UI/UX 设计语言
- 性能优化：快速响应，流畅交互
- 数据安全：密码等敏感信息加密存储

### 非目标
- 云端同步（第一阶段不考虑）
- 多用户支持（单用户应用）
- 实时协作功能
- 移动端支持（仅桌面端）

## 决策

### Decision 1: 数据库选择
**选择**: SQLite（通过 Tauri SQL Plugin）
**理由**: 
- 本地文件数据库，符合 Local First 原则
- Tauri 官方支持，集成简单
- 适合中小型数据量
- 无需额外服务

**替代方案**: 
- IndexedDB：浏览器原生，但 Rust 端访问复杂
- JSON 文件：简单但查询性能差

### Decision 2: Markdown 编辑器选择
**选择**: 评估 `milkdown` 或 `@toast-ui/vue-editor`
**理由**:
- 需要 Vue3 兼容
- 支持图片/视频嵌入
- 良好的扩展性

**替代方案**:
- `vue-markdown-editor`: 功能较简单
- `CodeMirror`: 更偏向代码编辑

### Decision 3: 存储架构
**选择**: 
- 笔记：文件系统（MD + 资源文件夹）
- 密码/书签/待办/日程：SQLite 数据库
- 用户设置：Tauri Store（JSON，用户目录）

**理由**:
- 笔记需要文件系统以便外部访问
- 结构化数据适合数据库
- 设置数据量小，Store 足够

### Decision 4: 插件系统设计
**选择**: 基于事件和接口的插件系统
**架构**:
```
插件接口 → 插件注册 → 插件生命周期 → 事件钩子
```

**理由**:
- 松耦合，易于扩展
- 支持动态加载
- 不影响核心功能

### Decision 5: AI 集成方式
**选择**: 前端直接调用 AI API（通过 HTTP）
**理由**:
- 简单直接，无需 Rust 端代理
- 支持多种 AI 服务
- 配置灵活

**替代方案**:
- Rust 端代理：增加复杂度，但可隐藏 API Key

### Decision 6: 布局结构
**选择**: 三栏布局
```
┌─────────────────────────────┐
│ HeaderBar (窗口控制)        │
├──────┬──────────────────────┤
│      │                      │
│ Side │   Content Area       │
│ bar  │   (Router View)      │
│      │                      │
└──────┴──────────────────────┘
```

**理由**:
- 清晰的功能分区
- 符合桌面应用习惯
- 易于导航

## 风险 / 权衡

### Risk 1: 数据迁移
**风险**: 未来版本升级可能需要数据迁移
**缓解**: 
- 设计版本化的数据库 Schema
- 实现迁移脚本
- 备份机制

### Risk 2: 插件系统复杂度
**风险**: 插件系统可能增加维护成本
**缓解**:
- 第一阶段实现基础插件接口
- 文档完善
- 提供示例插件

### Risk 3: 性能问题
**风险**: 大量数据可能影响性能
**缓解**:
- 分页加载
- 虚拟滚动
- 索引优化

### Risk 4: AI API 成本
**风险**: 频繁调用可能产生高成本
**缓解**:
- 用户配置 API Key
- 限制调用频率
- 缓存结果
